| 模块 | 引入方式 | 推荐度 | 优先级 |
|------|---------|--------|--------|

## 2. 逐模块利弊分析

### 2.1 记忆系统 (openmem)

**利:**
1. **跨 Session 连续性**: 代理记住项目架构/约定/坑, 下次对话直接利用
2. **渐进式理解**: 随使用次数增加, 记忆越来越准确
3. **低成本实现**: 纯插件, 无需改 OC core
4. **可迁移**: 记忆文件是纯 markdown, 可跨工具复用
5. **社区价值**: 这是 OC vs CC 最大的功能差距, 填补后差异化减小

**弊:**
1. **存储膨胀**: 长时间使用后记忆文件可能很大
2. **准确性风险**: LLM 提取/巩固可能引入错误信息
3. **Prompt 成本**: 每次注入 MEMORY.md 消耗 token
4. **维护负担**: 巩固 prompt 需要持续调优

**缓解:**
- MEMORY.md 限制 ≤25KB
- 记忆文件允许用户手动编辑纠正
- 提供 `/memory-clean` 命令清理

---

### 2.2 活文档 (magicdocs)

**利:**
1. **文档自动更新**: 代码变, 文档自动跟变
2. **低侵入性**: 仅编辑标记文件, 不影响其他文档
3. **安全**: 子 agent 权限极窄 (仅 Edit → 目标文件)
4. **灵活**: 支持自定义更新指引

**弊:**
1. **可能误更新**: LLM 可能理解错误并修改正确文档
2. **额外 API 调用**: 每轮结束后可能触发子 agent
3. **文件膨胀**: 如不限制, MAGIC DOC 可能越来越多

**缓解:**
- 子 agent 使用 cheap 模型 (small_model)
- 提供 git diff 预览, 用户可撤销
- 限制同时追踪的文档数

---

### 2.3 LSP 诊断投递 (lsp-diagnostics)

**利:**
1. **模型自动感知错误**: 无需每次手动查询 LSP
2. **减少幻觉**: 模型在写代码前就知道已有错误
3. **低实现成本**: OC 已有 LSP 基础, 只需添加诊断管道
4. **去重+限流**: 避免相同错误重复占用上下文

**弊:**
1. **可能干扰**: 大量诊断可能淹没有用的上下文
2. **延迟**: 诊断通知是异步的, 可能不反映最新文件状态
3. **仅支持已配置的 LSP 服务器**: 如用户未配置, 无效果

**缓解:**
- 音量限制 (≤10条/文件, ≤30条/总计)
- 文件修改后清除过时诊断
- 按严重性排序, 优先显示 Error

---

### 2.4 增强子代理 (deepagent)

**利:**
1. **后台任务**: 解放终端, agent 可并行工作
2. **Brief 工具**: 模型可主动汇报进度
3. **Task 系统**: 长期任务可跟踪/恢复

**弊:**
1. **后台任务复杂性**: 需要任务队列/状态管理
2. **资源竞争**: 多个子代理并行消耗 API 配额
3. **实现成本**: 需要 5+ 个新工具定义

**缓解:**
- 限制同时运行的子代理数
- 使用 small_model 执行后台任务
- Brief 工具的消息有摘要属性避免 spam

---

### 2.5 迭代式规划 (deep-plan)

**利:**
1. **更灵活的规划**: 允许多次修改而非一次性通过
2. **减少执行错误**: 完善计划再执行, 减少返工
3. **低实现成本**: 主要是 skill prompt 增强

**弊:**
1. **增加交互轮次**: 可能需要多次拒绝→修改循环
2. **用户阻塞**: 仍需等待 plan 完成

**缓解:**
- 设置最大迭代次数 (如 3 次)
- 后续 Phase 2 可结合异步 plan

---

### 2.6 远程执行 (remote-agent)

**利:**
1. **解放终端**: 长任务在远程运行
2. **更强硬件**: 可在 GPU/大内存机器上运行
3. **任务恢复**: 断连后可恢复
4. **团队共享**: 多人可访问同一执行环境

**弊:**
1. **高实现成本**: 需要 SSH/Docker 桥接 + 状态同步
2. **安全风险**: 远程执行面大, 需要沙箱
3. **网络依赖**: 延迟影响实时性
4. **环境一致性**: 远程环境与本地可能不同
5. **维护成本**: SSH/Docker/网络问题排查困难

**缓解:**
- MVP 用简单 SSH, 不做 Docker 沙箱
- Git bundle 上传代码确保一致性
- 支持超时 + 断连恢复

---

### 2.7 桌面自动化 (computer-use) — 不建议引入

**利:**
1. 测试自动化: 可做端到端 UI 测试
2. 演示录制: 自动操作桌面应用

**弊:**
1. **范围偏差**: 超出编码代理定位
2. **仅 Darwin**: 不支持 Linux/Windows
3. **极高安全风险**: 全局输入/截屏
4. **极高实现成本**: Swift+Rust+CFRunLoop 泵送
5. **极高维护成本**: 随 macOS 版本迭代
6. **有替代方案**: Playwright MCP 覆盖 web 自动化

**结论: 不建议引入 core, 可作为独立 MCP 服务器供特定用户使用。**

## 3. 风险矩阵

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 记忆系统误写入 | 中 | 中 | 用户手动编辑 + git 追踪 |
| LSP 诊断洪水 | 中 | 中 | 去重+限流 |
| 后台任务失控 | 低 | 高 | 最大并发限制, 超时终止 |
| 远程执行安全事件 | 低 | 高 | 沙箱 + 最小权限 |
| 插件兼容性 break | 中 | 中 | OC 版本锁定, 测试 CI |
| API 成本失控 | 中 | 中 | small_model + 频率限制 |

## 4. 实施建议

### Phase 1 (MVP, 2-4周)
1. openmem: 即时提取 + 手动巩固
2. magicdocs: MAGIC DOC 检测 + 自动更新
3. lsp-diagnostics: 诊断投递管道

### Phase 2 (增强, 4-8周)
4. deepagent: Brief + Sleep + 后台 Task
5. openmem: 自动巩固 (autoDream)
6. deep-plan: 迭代式规划 Skill

### Phase 3 (高级, 8-16周)
7. deepagent: Task CRUD 系统
8. remote-agent: SSH 远程执行

### Phase 4 (实验, 后续)
9. computer-use-mcp: 独立 MCP 项目
10. task-scheduler-mcp: Cron 定时任务
